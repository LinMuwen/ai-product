## 一、交互方式：AI对话式界面

### 1.1 整体交互形态

产品核心交互为**AI对话式聊天界面**，参考 ChatGPT / 微信聊天 的交互模式。用户通过自然语言与AI成长顾问持续对话，完成从"规划目的选择"到"路径推演"到"任务跟踪"的全流程。

### 1.2 部署形态

| 终端 | 形态 | 说明 |
|------|------|------|
| Web端（PC） | 响应式网页 | 桌面浏览器访问，左侧对话区 + 右侧数据看板（可收起） |
| 移动端 | PWA / H5 | 手机浏览器访问，聊天全屏 + 底部输入栏；支持系统级推送通知 |

### 1.3 对话式功能映射

对话不只是文字问答，需在聊天流中嵌入交互组件：

| 场景 | 聊天流中的表现 | 开发要点 |
|------|---------------|---------|
| **目的选择** | AI发送一条消息，内含选项卡片（自我实现 / 经济独立 / 摆脱原生家庭 / 体验多元人生），用户点击卡片即完成选择 | 消息类型支持 `message_type: "option_card"`，卡片可单选 |
| **量表测评** | AI逐题发送量表题目，每题一条消息，用户点击选项按钮作答。进度小字显示"第3/20题" | 消息类型支持 `message_type: "scale_question"`，携带题号、选项列表、已答题数 |
| **画像报告** | AI推送一条消息，内含可展开的画像雷达图卡片（ECharts渲染），点击可全屏查看 | 消息类型支持 `message_type: "chart_card"` |
| **路径推演** | AI发送路径对比卡片，三赛道并排展示（难度/时间成本/匹配度），用户点击某赛道进入详情 | 消息类型支持 `message_type: "path_compare"` |
| **任务清单** | AI发送结构化任务列表消息，每项含复选框 + 截止日期 + 完成标准，勾选即反馈进度 | 消息类型支持 `message_type: "task_list"` |
| **可穿戴设备数据报告** | AI在检测到异常生理信号后主动推送一条消息："我注意到你这周三下午压力值偏高，当时发生了什么？"，附带快捷回复选项（"在备考" / "在实习" / "个人事务" / "输入其他"） | 消息类型支持 `message_type: "stress_probe"`，触发条件为可穿戴数据异常检测 |

### 1.4 消息类型数据结构定义

```json
{
  "message_id": "uuid",
  "role": "assistant | user | system",
  "timestamp": "ISO8601",
  "message_type": "text | option_card | scale_question | chart_card | path_compare | task_list | stress_probe",
  "content": { /* 根据 message_type 不同，content 结构不同 */ },
  "meta": {
    "conversation_context": "purpose_selection | assessment | path_exploration | task_tracking",
    "trigger_event": "user_input | scheduled_check | biosignal_anomaly | task_deadline"
  }
}
```

### 1.5 手机弹窗（Push Notification）

| 触发条件 | 推送内容示例 | 点击行为 |
|---------|------------|---------|
| 阶段任务临近截止日 | "你的《行测言语理解模考》任务还剩2天，当前进度0% → 点击查看" | 打开H5 → 进入该任务的对话上下文 |
| 可穿戴设备检测到异常压力 | "检测到你这周有2次高压力事件，我们聊聊？" | 打开H5 → AI主动发起 stress_probe 对话 |
| 每周复盘提醒（周日20:00） | "本周复盘时间到！看看你离目标还有多远？" | 打开H5 → 展示本周进度卡片 |
| 赛道关键节点预警 | "考研预报名将在10天后开始，你准备好了吗？" | 打开H5 → 展示该节点的checklist |

## 三、规划完成情况检测

### 3.1 设计目的

系统需要**持续监控**学生是否偏离规划路径，并在偏差发生时主动介入。这不是一次性报告，而是一个持续的"规划健康度"体检系统。

### 3.2 检测维度

| 检测维度 | 检测内容 | 触发条件 | 响应方式 |
|---------|---------|---------|---------|
| **任务完成率** | 本月/本学期任务完成比例 | 连续2周完成率 < 50% | AI主动对话："最近任务进度偏慢，是遇到什么困难了吗？" |
| **目标偏离度** | 画像特征变化是否与赛道要求背离 | GPA下降>0.3 / 连续3周未学习相关课程 | AI提示风险，并建议是否需要切换赛道或调整计划 |
| **赛道匹配度漂移** | 定期重新计算画像→赛道匹配度 | 每学期末自动重算，匹配度下降>15% | 推送新画像报告+路径重新推演建议 |
| **节点预警** | 关键时间节点（报名截止/考试日期等） | 距离节点7天/3天/1天三级预警 | Push通知，对话流中置顶提醒 |
| **身心状态预警** | 可穿戴数据异常 | 单周3次以上应激事件 / 睡眠质量持续<6小时 | AI推送 stress_probe 消息，进行原因归因对话 |

### 3.3 检测结果交互

所有检测结果以**对话消息**形式呈现，不要独立弹窗或系统通知：

```
AI: 本周检测报告（7.10-7.17）

📊 任务完成率：60%（低于基准线 80%）
    - 已完成 3/5 项周任务
    - 未完成：《数据结构第4章习题》（已逾期2天）

⚠️ 赛道匹配度：稳定（公考赛道 87%）
    - 较上期末无显著变化

💚 身心状态：正常
    - 本周无应激事件记录，睡眠均值 7.2h

💡 建议：本周围绕"拖延"聊一下？你可以告诉我最近是什么阻碍了你。
                                                 [聊聊看] [先不管]
```

## 四、数据持续采集：情绪事件监测

### 4.1 设计目的

传统的职业测评是"填一次问卷管四年"，但我们认为：**一个人的抗压能力、情绪模式、精力规律只有通过持续监测才能真实还原。** 可穿戴设备（智能手表/手环）提供了一种无感、持续的数据采集方式。

### 4.2 采集指标

| 指标 | 采集频率 | 数据来源 | 用途 |
|------|---------|---------|------|
| 心率（HR） | 5分钟/次 | 手环/手表 | 压力事件检测基础 |
| 心率变异（HRV） | 5分钟/次 | 手环/手表 | 自主神经状态，压力水平核心指标 |
| 静息心率（RHR） | 每天1次 | 手环/手表 | 长期压力趋势 |
| 睡眠时长与质量 | 每天1次 | 手环/手表 | 恢复能力、精力阈值 |
| 活动/步数 | 持续 | 手环/手表 | 日常活跃度辅助判断 |
| 皮肤电反应（EDA） | 有条件采集 | 部分手表 | 情绪唤醒度（加分项，非必须） |

### 4.3 应激事件检测算法（简化版）

```python
# 压力事件判定逻辑（可调参）

def detect_stress_event(hrv_data, hr_data, time_window_min=30):
    """
    当满足以下任一条件时，标记为"应激事件"：
    1. HRV连续15分钟低于个人基线40%以上
    2. 心率连续15分钟高于静息心率30%以上（排除运动时段）
    """
    events = []
    for window in sliding_window(hrv_data, time_window_min):
        if window.avg_hrv < baseline_hrv * 0.6:
            events.append(StressEvent(
                start_time=window.start,
                end_time=window.end,
                severity=calc_severity(window),
                hr_peak=max(window.hr),
                hrv_min=min(window.hrv)
            ))
    return events
```

### 4.4 应激事件→对话归因闭环

当系统检测到应激事件后，触发以下交互流程：

```
Step 1：缓冲判定
  - 单次事件：加入待确认队列，不立刻推送（避免频繁打扰）
  - 24h内≥3次事件 或 单次严重度≥0.8：立刻触发推送

Step 2：AI主动推送
  AI发送 stress_probe 消息：
  "我注意到你昨天下午3:00-3:45压力值很高（心率峰值102bpm），当时你在做什么？"
  快捷选项：[在备考复习] [在实习工作] [在开会/社交] [在写论文] [其他...]

Step 3：用户选择/输入
  - 用户点击选项 → 系统记录标签
  - 用户选择"其他" → 文本输入框出现

Step 4：AI分析 + 数据库查询
  - 如果是重复标签（如过去30天有5次"备考复习"触发应激）：
    AI："过去一个月你有5次备考期间出现高压力反应，我查了一下案例库，
         有类似应激模式的人中，65%最终选择了压力更低的岗位方向。
         你觉得需要重新评估一下公考这条路的适配度吗？"
  - 如果是新的孤立事件：
    AI记录为观察项，不做过度解读

Step 5：画像更新
  - 应激标签写入用户画像的"生理指标→情绪事件"维度
  - 用于后续赛道匹配时的"岗位风险标记"
```

### 4.5 数据隐私与安全

```
1. 所有原始生理数据仅存储在用户本地设备，不上传服务器
2. 仅上传脱敏后的"事件摘要"：时间窗口、应激等级、用户自报的活动标签
3. 用户可随时在设置中关闭生理数据采集
4. 数据用于个人规划辅助，不做任何其他用途
```

### 4.6 开发实现路径（比赛阶段）

由于比赛阶段无法获取真实可穿戴设备数据，建议采用以下替代方案：

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| **方案A：模拟数据流** | 预置10组典型学生的"一周生理数据CSV"，包括HR/HRV/睡眠/应激事件标签，Web demo读取并模拟推送 | 可完整演示闭环，开发成本低 | 数据不真实 |
| **方案B：用户手动录入** | 在demo中提供一个"模拟应激事件"按钮，点击后触发完整对话归因流程 | 评审时可互动演示 | 体验不够流畅 |
| **推荐：A+B组合** | demo默认加载一组模拟数据自动触发stress_probe；同时提供"模拟新事件"入口供评委手动触发 | 兼顾自动演示和交互灵活性 | — |

## 五、附录：开发优先级建议

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | 对话式界面核心框架 + 消息类型系统 | 一切交互的基础，先跑通 | 
| P0 | 六维画像采集流程 | 数据入口，没有画像后面全跑不了 |
| P0 | 三层推演核心逻辑 | 产品核心差异化，必须可演示 |
| P1 | 游戏化进度条 | 用户体验亮点，评审加分项 |
| P1 | 规划完成情况检测 | 体现"持续陪伴"理念的关键功能 |
| P2 | 情绪事件监测（模拟数据版） | 创新点展示，但依赖可穿戴数据模拟 |
| P2 | 手机Push通知 | 锦上添花，比赛现场可先做H5内模拟 |
